Architecture MVC/VM/P

Acronymes respectifs de « Modèle Vue Contrôleur/ViewModel/Présentation », ces trois appellations désignent en réalité un mode de construction d’applications similaire. Il s’agit de diviser les différentes pages de code qui composent une application en trois catégories.

  • La première catégorie, le modèle, sert à définir, créer, récupérer et modifier les données qui seront utilisées dans l’application.
  • La seconde catégorie, la vue, se compose des différentes fenêtres et écrans qui serviront à l’utilisateur pour manipuler ladite application, et les données qui la compose.
  • Enfin, la troisième partie, qui peut prendre le nom de Contrôleur, ViewModel ou encore Présentation, sert d’intermédiaire entre la vue et le modèle, transformant les données pour les faire s’intégrer dans l’une ou l’autre de ces catégories.

J’ai tout d’abord appris à utiliser ce type d’architecture sous le nom de MVC, lors de ma première année d’études. Plus tard, lorsque nous travaillons sur le projet Hermès, j’appris que cette architecture portait le nom de MVVM pour les technologies Android. Enfin, lors de mon entrée en entreprise et de mon travail sur le projet Portail R&R, je vis que cette architecture portait encore un autre nom pour les technologies WinDev : MVP.
